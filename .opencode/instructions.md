# Repository Engineering Instructions

This file defines engineering rules for agents working in this repository.

Use this file for universal engineering law that applies regardless of
tech stack. Stack-specific rules (languages, frameworks, testing tools,
build commands) are defined in `REPOSITORY-CONTEXT.md` and optionally
in `.opencode/instructions-stack.md`.

- `AGENTS.md` defines the delivery cell charter
- `REPOSITORY-CONTEXT.md` defines current repository facts and stack rules
- `.opencode/instructions-stack.md` defines stack-specific engineering rules (generated by `/setup`)
- this file defines how work must be done in any repository using this agent system

---

## Scope

These instructions apply to:
- context mapping
- spec writing
- implementation
- review
- documentation updates where engineering rules matter

If guidance conflicts, follow the precedence defined in `AGENTS.md`.

---

## Repository structure

Agents must:
- preserve the existing repository structure once the repo is customized
- prefer extending established module, feature, and project boundaries
- avoid introducing parallel structures for the same concern
- escalate when the repository shape is unclear or incomplete

Repository-specific paths, commands, and architecture facts belong in
`REPOSITORY-CONTEXT.md`.

---

## Architecture

### Layer separation

Regardless of stack, enforce the architecture boundaries defined in
`REPOSITORY-CONTEXT.md`.

Rules:
- controllers, handlers, or API endpoints should contain orchestration only, not business logic
- domain or business logic must not depend on infrastructure concerns
- infrastructure may implement interfaces or contracts defined in domain layers
- do not introduce circular dependencies
- do not move logic across layers unless explicitly required by the task
- preserve established project or module segregation unless the user explicitly approves an architecture change

### Cross-stack changes

If the task spans multiple layers or stacks (e.g., backend and frontend):
- keep contracts aligned
- update affected consumers when producer contracts change
- do not silently change API or interface behavior without updating relevant consumers and tests
- call out versioning, migration, or rollout risks when the impact crosses boundaries

---

## Implementation

### Scope control

Implementers must:
- modify only approved files and scope
- keep changes small and reviewable
- avoid unrelated cleanup, renames, or refactors
- justify any new dependency or major pattern change

Do not:
- redefine requirements
- change public contracts unless explicitly required
- mix feature work with unrelated maintenance

### Language and framework rules

Follow the language-specific and framework-specific rules defined in:
1. `.opencode/instructions-stack.md` (if it exists)
2. `REPOSITORY-CONTEXT.md` stack conventions section

If neither file defines rules for the relevant language or framework,
follow the existing patterns visible in the repository and escalate
when conventions are unclear.

### Package manager discipline

Use the package manager defined in `REPOSITORY-CONTEXT.md`.
Do not introduce alternative package managers, lockfiles, or workflow
guidance unless the user explicitly changes the repository standard.

---

## Testing

### Universal expectations

Rules:
- every new public method, endpoint, or exported function must have at least one test
- changed behavior must be covered by tests
- bug fixes should include regression coverage where practical
- keep tests behavior-focused, not implementation-coupled

### Stack-specific testing

Use the testing frameworks and tools defined in `REPOSITORY-CONTEXT.md`.

Do not:
- add excessive mocking where simpler behavior tests are possible
- claim test coverage that was not executed
- treat pre-existing failing tests as newly introduced defects

---

## Build and validation

Rules:
- use the build and test commands defined in `REPOSITORY-CONTEXT.md`
- validate in the intended working directory, not by assumption
- distinguish backend, frontend, and full-repo validation when reporting results

Implementers must:
- verify the baseline build before changes for non-trivial work when practical
- run the relevant tests for the changed area
- report what was run and what was not run

Reviewers must:
- distinguish verified results from unverified assumptions
- note when correctness depends on code paths not exercised

---

## Data and persistence

Rules:
- use the persistence technology and patterns defined in `REPOSITORY-CONTEXT.md`
- preserve existing data access patterns and boundaries
- do not bypass repository conventions for persistence without explicit reason
- schema-affecting changes must be treated as high impact
- shared contracts and persistence models with cross-workflow impact must be escalated when scope is unclear

---

## Known issue handling

The repository may have pre-existing issues listed in `REPOSITORY-CONTEXT.md`.

Rules:
- do not treat listed pre-existing warnings or failures as regressions
- do not fix nearby issues unless the task explicitly includes that work
- distinguish new defects from known baseline noise in implementation and review reports

---

## Documentation

Rules:
- document current behavior, not intended future behavior
- update documentation when code or configuration changes materially affect how the repository operates
- update `REPOSITORY-CONTEXT.md` when repository facts relevant to agents change
- preserve existing document structure unless explicitly asked to reorganize it

---

## Review

Reviewers must validate changes against:
- approved scope
- architecture boundaries
- testing expectations
- security-sensitive paths where relevant
- unintended scope expansion

Reviewers should:
- separate blocking issues from suggestions
- identify what was verified directly
- avoid requesting unrelated improvements outside task scope unless correctness, security, or architecture require it

---

## Escalate when

Escalate if work appears to require:
- layer boundary violations
- shared contract changes with broad impact
- new architectural patterns
- new infrastructure assumptions
- security-sensitive flow changes
- ambiguous workflow, persistence, or integration changes with cross-module impact

---

## Project setup

When this template is used for a new repository, run `/setup` to configure
the agent system for your specific tech stack. The setup command generates:
- `REPOSITORY-CONTEXT.md` with correct paths, commands, and conventions
- `.opencode/instructions-stack.md` with language and framework-specific rules
- Agent bash permission updates for your build tools

Run `/setup` again when the stack changes materially.

---

## Final principle

Follow repository reality and existing patterns.

When in doubt:
- stay within approved scope
- preserve layer boundaries
- reuse existing patterns
- test changed behavior
- update repository memory when facts change
- escalate before making broad architectural moves
